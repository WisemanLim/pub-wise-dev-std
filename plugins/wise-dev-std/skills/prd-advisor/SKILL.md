# PRD Advisor — PRD 작성 도우미 지식 베이스

이 스킬은 PRD 작성을 **설문 → 초안 생성 → 리뷰 체크** 3단계로 자동화하는 로직과 근거를 담는다.
근거 문서: `${CLAUDE_PLUGIN_ROOT}/../../.doc/Perplexity-PRD 작성을 위한 기법 조사 및 비교, 추천.md`.

핵심 원칙(문서 §4): **AI 는 작성자가 아니라 보조 브레인**이다. 문제 정의·판단은 사용자가 하고,
스킬은 구조·초안·누락 점검을 맡는다. 추측한 값은 반드시 `가정:` 꼬리표로 표시한다.

산출물은 프로젝트 루트 `PRD.md` 한 개. 이후 `/wise-dev-std:recommend` 가 이 `PRD.md` 를
읽어 스택 × 업종 추천으로 이어진다(흐름 4-1 의 1번 "PRD.md 를 둔다" 를 자동화).

## 0. 작업 순서 (always)

1. 루트에 이미 `PRD.md`/`prd.md`/`docs/PRD.md` 가 있으면 → **덮어쓰지 않는다**. 보강/갱신할지 사용자에게 묻는다.
2. §2 의 5 핵심 질문을 **한 번에 묶어** 사용자에게 제시(비기술 직군도 답할 수 있는 자연어).
3. 업종 키워드가 답에 있으면 §5 로 KSIC 도메인 오버레이를 해석(`${CLAUDE_PLUGIN_ROOT}/domains/*.yaml`),
   그 업종의 성공지표·비기능요구사항 후보를 **제안**한다(불명/순수 SW = `ict-saas` 베이스라인).
4. §4 규칙으로 §3 One-Page 템플릿을 채워 `PRD.md` 초안을 생성. 모르는 값은 `가정:` 표시.
5. `--full` 이면 §6 의 풀스펙 확장 섹션을 덧붙인다.
6. §7 리뷰 체크리스트로 자가 점검 결과를 한 줄 보고.
7. 마지막 안내: "`/wise-dev-std:recommend` 로 이 PRD 기반 스택·업종 추천을 이어가세요."

## 1. 기법 선택 (문서 §1·§5)

상황별 권장 조합. 기본은 **One-Page**, 복잡/규제 도메인은 `--full` 로 확장한다.

| 상황 | 권장 | 이 스킬 동작 |
|---|---|---|
| 신규 SaaS/앱 MVP | One-Page + 문제중심 + What-Why-How | 기본 설문 |
| 대형 엔터프라이즈(6주+) | 2단계: One-Page → 풀스펙 | `--full` |
| 규제·보안·데이터 거버넌스 강함 | 풀스펙 + Living PRD | `--full` + 업종 오버레이 |
| 짧은 실험/AB 테스트 | Lean + 문제중심 | 기본 설문(성공지표 강조) |

구조 프레임은 **What–Why–How + Success**(측정)로 고정한다. Why 는 문제중심으로 강화
(VOC·로그·근거 데이터를 기술하도록 유도).

## 2. 핵심 설문 (5질문 — 비기술 직군 정렬용, 문서 §1·§2·§4)

비기술 직군도 답할 수 있는 "문제–사용자–범위–경험–성공" 프레임. 한 번에 묶어 질문한다.

1. **Why (왜)** — 해결하려는 핵심 문제는? 왜 지금 중요한가? (근거 데이터/VOC/로그 있으면 함께)
2. **Who (누구)** — 대표 사용자/역할과 대표 사용 상황은?
3. **What (무엇)** — 이번 릴리즈 핵심 기능 3~5개 + 이번엔 **하지 않을 것(Out of Scope)**?
4. **How (어떻게 보이나)** — 대표 사용 흐름 1~2개를 화면/단계 순서로?
5. **Success (성공)** — 출시 후 무엇을 보면 성공? 가능하면 지표·기준선·목표값으로.

부족하면 추가로: 프로젝트 이름, **업종(KSIC 대분류)**, 주요 리스크/의존성/미결정 사항.

> 확장 10질문(`--full` 시): 위 5개 + 우선순위 근거(P0/P1), 측정 방법(로그/이벤트), 기술 제약,
> 외부 연동/인터페이스, 데이터·개인정보 처리 요구. (문서 §1 Step 1~5)

## 3. One-Page PRD 템플릿 (문서 §2·§3 — 자동 생성용)

```md
# One-Page PRD: {{프로젝트 이름}}

## 1. 개요
- 한 줄 요약:
- 배경 / 왜 지금 하는가:
- 작성자 / 날짜 / 버전:

## 2. 문제와 대상 (Why & Who)
- 해결하려는 핵심 문제:
- 비즈니스 관점 중요성:
- 근거 데이터 / VOC / 로그:
- 타깃 사용자 / 페르소나:
- 대표 사용 상황:

## 3. 범위 (What — In / Out)
- 이번 릴리즈 핵심 기능 (최대 5개, P0/P1):
- 하지 않을 것 (Out of Scope):

## 4. 사용자 경험 (How — User Flow)
- 대표 시나리오 1: "누가, 어떤 상황에서, 무엇을 하려고, 어떤 흐름으로"
- 대표 시나리오 2 (선택):

## 5. 성공 지표 (Success)
- 핵심 KPI / 기준선(Baseline) / 목표값(Target) / 측정 방법:

## 6. 리스크 및 메모
- 주요 리스크:
- 의존성 (내부/외부):
- 미결정 사항 (Open Questions):
```

## 4. 생성 규칙 (문서 §1·§3 표)

| 항목 | 최소 규칙 |
|---|---|
| 문제 정의 | "문제–영향–근거" 구조, 3문장 이내 |
| 목표 | 비즈니스 목표와 사용자 가치를 분리 서술 |
| 성공 지표 | 기준선·목표값·측정법을 **모두** 포함 (모르면 `가정:`) |
| MVP 범위 | In / Out Scope **둘 다 필수** — 빠지면 리뷰 차단 |
| 요구사항 | 기능/비기능 분리, 수용 기준은 Given-When-Then 권장 |
| 언어 | 전문용어는 일상 언어로 먼저 쓰고 괄호에 기술용어 (이중 레이어, 문서 §5) |

각 섹션 5~7줄 이내. 표가 적절하면 표 사용. 모르는 수치는 `가정:` 으로 명시.

## 5. 업종(KSIC) → 성공지표·NFR 자동 제안 (domains/*.yaml 연동)

설문 답에 업종 키워드가 있으면, 그 업종의 `domains/<id>.yaml` 을 읽어 PRD 후보를 채운다.
매핑은 stack-advisor §1.5(KSIC 대분류 → 도메인 오버레이)를 그대로 사용. 불명 = `ict-saas`.

도메인 YAML 에서 다음을 PRD 후보로 끌어온다(교체가 아니라 **제안**, 사용자가 확정):

- `korea_regulations` / `global_compliance` → 6. 리스크 + (풀스펙 시) 비기능 요구사항의 **규제** 항목
- `data_classes` → (풀스펙) 데이터·개인정보 처리 요구사항
- `testing_additions` → 5. 성공 지표의 **검증 가능한 수용 기준** 후보
  (예: 금융=멱등결제/정산대사, 의료=비식별 검증·FHIR 적합성, 커머스=오버셀 방지, 게임=확률공개 audit)
- `infra_patterns` / `stack_overrides` → (풀스펙) 기술 메모의 제약

> 도메인 데이터는 이미 존재하는 자산이다. 새로 추측하지 말고 YAML 근거 + 출처(`references`)를 인용한다.

## 6. 풀스펙 확장 (`--full` — 문서 §4 "2단계 PRD", 기술 리더용)

One-Page 승인을 전제로 아래 섹션을 덧붙인다. 비기술 요약(1~6)과 기술 메모를 **명확히 분리**한다.

```md
---

## 7. 기술 메모 (Technical Notes)
- 주요 기술 결정 (아키텍처·핵심 컴포넌트·인터페이스):   # 예시/초안임을 명시
- 비기능 요구사항 (성능·보안·규제·SLO·Observability):   # 업종 오버레이 반영
- 데이터 / 로깅 요구사항 (지표 측정 관점):
- 외부 연동 / API 스펙 요약:
- 리스크 및 트레이드오프:

## 8. 실행 연결 (Epic / Story 힌트)
- 구현을 위해 나뉘어야 할 대략적 Epic/Work Item 후보:
```

규칙: 섹션 1~6 은 비기술 언어, 7~8 은 개발 리더 밀도. 구현 방식은 "예시/초안" 명시.

## 7. 리뷰 체크리스트 (Definition of Ready — 문서 §3)

생성 후 자가 점검(한 줄 보고). 미충족 항목은 `가정:`/`Open Question` 으로 남긴다.

- [ ] 문제 정의가 실제 사용자 문제를 설명하는가?
- [ ] 이번 MVP 범위(In)와 비범위(Out)가 둘 다 명확한가?
- [ ] 성공 지표가 측정 가능하고 기준선이 있는가?
- [ ] 수용 기준이 테스트 가능한 문장(Given-When-Then)인가?
- [ ] 의존성·리스크가 숨겨져 있지 않은가?
- [ ] (업종 적용 시) 규제·데이터등급이 비기능 요구사항/리스크에 반영됐는가?

## 8. 출력 후 안내

`PRD.md` 생성 후 항상:

> `PRD.md` 초안을 만들었습니다. 검토·수정 후
> `/wise-dev-std:recommend` 로 PRD 기반 스택 × 업종 추천을 이어가세요.
> (업종을 입력했다면 `/wise-dev-std:recommend <언어> <업종>` 형태로 힌트를 줄 수 있습니다.)
